### [P0] Post-task refetch clobbers concurrent edits; failed render lets stale local state clobber render progress
- 位置:`public/app.js:342` · 来源:frontend-ux · 置信:high
- 证据:runOp unconditionally replaces local state after any task: `const fresh = await api(.../storyboard)...; if (fresh) { state.sb = fresh; renderBoardHeader(); renderScenes(); }` (app.js:341-342). Textareas stay editable during tasks — setOpsDisabled (app.js:347) only disables .op/.render-one buttons. The 409 retry path (app.js:321-324) keeps edits local and reschedules save(5000). On task FAILURE, pollTask throws before the refetch, so state.sb keeps the pre-render copy lacking the render's per-scene renderedPath/status/renderedHash writes; the pending 5s retry then PUTs that stale copy over disk (server only 409s while the render task is still queued/running, server.ts:108-114).
- 后果:Success path: editing any scene while a render runs triggers a 409-deferred save ("渲染中,稍后自动保存"); when the render finishes, runOp's wholesale refetch (app.js:341-342) erases the typed text from screen and memory, and the still-pending 5s retry then saves the edit-free storyboard as "已保存" — silent loss of user edits that the UI explicitly promised to auto-save. A race variant lets the retry fire between task completion and refetch, clobbering the render's disk writes instead. Failure path: after a partial render failure there is no refetch, so the pending retry or any subsequent edit PUTs the pre-render copy, wiping renderedPath/renderedHash/status on disk for completed scenes — UI shows them 待渲染 and failure badges never appear. Correction to original impact: the wiped hashes do NOT directly force re-rendering (cache is content-addressed and self-heals on the next full render, render.ts:984-991); instead, a subsequent successful single-scene render runs pruneStaleSceneArtifacts (render.ts:1095) with only that scene's hash live and deletes the completed scenes' cached mp4s from disk (render.ts:82-93) — real artifact loss, after which those scenes genuinely re-render from scratch.
- 修法:Track a dirty flag (set in scheduleSave, cleared on successful save). In runOp: always refetch (also on failure), but instead of wholesale `state.sb = fresh`, merge only render-owned fields (per-scene renderedPath, renderedHash, status; stages; project untouched) into state.sb when dirty, and only replace wholesale when clean. Cancel the pending retry timer before merging so the merged sb is what gets saved.
- 验证:Every load-bearing claim checks out in the code. (1) app.js:341-342: after ANY task, runOp wholesale-replaces local state — `const fresh = await api(`/api/projects/${state.pid}/storyboard`).catch(() => null); if (fresh) { state.sb = fresh; renderBoardHeader(); renderScenes(); }` — and renderScenes() (app.js:94-99) rebuilds all textareas from state.sb, destroying unsaved typed text. (2) app.js:347: `setOpsDisabled` only touches `.op, .render-one` buttons; textareas (app.js:182-187) and selects (208-216) stay editable during tasks and call scheduleSave. (3) app.js:321-324: on 409 the client keeps edits local, shows "渲染中,稍后自动保存", and schedules `state.saveTimer = setTimeout(save, 5000)`; runOp never cancels this timer. (4) server.ts:586-597: PUT wholesale-replaces storyboard.json; the guard `projectRenderBusy` (server.ts:108-114) returns false the instant the render task reaches "failed"/"succeeded" (server.ts:348-357 marks failed on throw). (5) render.ts:1025-1031/1050-1054/1078 write per-scene renderedPath/renderedHash/status incrementally to disk; render.ts:1084-1090 throws when any scene failed, so the task fails AFTER the disk copy holds completed scenes' state, and app.js:356 (`if (t.status === "failed") throw`) makes runOp skip the refetch — state.sb stays pre-render. Success-path loss is reachable in completely normal use (the 409 message explicitly invites the user to keep editing, then the refetch silently discards the edit and the pending retry reports "已保存" for an edit-free save). Failure-path clobber is equally real: any post-failure save (pending retry or next edit) PUTs the stale pre-render copy, wiping renderedPath/renderedHash/status for completed scenes on disk. One impact detail is inaccurate: the claim "the hash cache is lost so they re-render from scratch" — the cache is content-addressed (render.ts:970-993, `scene.{srcHash}.mp4` found by recomputing the hash; storyboard self-heals at 989-991), so a wiped renderedHash alone does NOT force re-rendering on the next full render. But an indirect, worse path exists: after the wipe, a successful single-scene render (`only`) reaches pruneStaleSceneArtifacts (render.ts:1095) with liveHashes containing only the just-rendered scene's hash (non-target scenes are skipped at render.ts:955-961 and never healed), so render.ts:82-93 rmSync-deletes the completed scenes' cached mp4s/dirs — real artifact loss, after which re-render from scratch is genuine. No QA gate covers saves (validate/review gate rendering only), so both paths are silent. Severity P0 stands: silent loss of user-typed content with a false "已保存" confirmation, plus on-disk state clobbering that can cascade to deletion of rendered artifacts, all reachable in ordinary single-user use.

### [P1] Storyboard write race: busy guard only covers render/stitch — PUT /storyboard during analyze/images/matte and POST /plan (which bypasses the FIFO queue entirely) silently lose user edits
- 位置:`src/server.ts:111` · 来源:render-core,server-security · 置信:high
- 证据:projectRenderBusy (server.ts:108-114) only treats op === "render" || op === "stitch" as busy, and the PUT /storyboard handler (server.ts:586-597) uses it as its sole conflict check — no version/updatedAt/If-Match field is compared. But analyze, images, and matte tasks also rewrite storyboard.json: runAnalyze reads the storyboard at task start (analyze.ts:378) and writes its whole in-memory copy at task end (analyze.ts:520); same pattern in images.ts:232, matte.ts:131/172, research.ts:202, and the render task rewrites it per scene (render.ts:1031/1054/1078). Additionally POST /plan runs runPlan synchronously in the HTTP handler, writing output/storyboard.json outside the task chain with no busy check at all (server.ts:563-577). Meanwhile the UI leaves textareas/selects/row-controls editable during ops (setOpsDisabled at public/app.js:347 only disables .op/.render-one buttons) and autosaves via debounced PUT (public/app.js:307-329).
- 后果:User edits scene text / deletes a scene while a minutes-long analyze/images/matte/research task runs (the UI leaves all fields editable and reports 已保存 after a 200 PUT). When the task finishes it writes back the whole storyboard it read at task start (analyze.ts:378→520, images.ts:157→232), silently erasing the saved edit; the UI refetch (app.js:341-342) then makes the edit visibly vanish with no error. Reverse interleaving instead discards the task's generated results. Additionally POST /plan (server.ts:563-577) bypasses both the FIFO chain and the busy guard, so a plan issued via API/CLI during a running render is silently reverted by the render's next per-scene storyboard write (render.ts:1031/1054/1078) — though the web UI itself only calls /plan at project creation (app.js:382), so that leg is not reachable by a normal UI click.
- 修法:Generalize the guard: projectTaskBusy(projId) = any queued/running task for the project (all ops mutate storyboard.json), and apply it to both PUT /storyboard and POST /plan (or route /plan through the FIFO chain like every other mutation) — same 409 path the render case already uses. Alternatively/additionally add a monotonic rev/updatedAt field to storyboard.json checked on PUT (reject stale writes with 409; the frontend already handles 409-and-retry).

### [P1] Watchdog/crash mid-encode leaves a truncated scene.<hash>.mp4 that poisons the content-addressed cache forever
- 位置:`src/render.ts:984` · 来源:render-core · 置信:high
- 证据:Cache hit is existence-only: `if (!opts.force && fs.existsSync(mp4Path))` (render.ts:984). But both engines write DIRECTLY to the final cache path: hyperframes `render --output <mp4Path>` (render.ts:156-162) and the Remotion color-normalize ffmpeg pass `-o <outMp4>` (render.ts:300-312). ffmpeg writes mp4 progressively (moov at end). The watchdog SIGTERM→SIGKILLs the process group on timeout (proc.ts:91-98), and daemon death orphans/abandons children — either leaves a partial file at the content-addressed path. fsutil.writeFileAtomic exists precisely for existence-based caches (fsutil.ts:4-10) but is not used here.
- 后果:A scene render timing out mid-encode (the watchdog's normal job) or a daemon crash/Ctrl-C during the mux window leaves a partial scene.<hash>.mp4 at the content-addressed cache path. Every subsequent render treats it as a cache hit (render.ts:984), overwrites the scene's 'failed' status back to 'rendered' (render.ts:989-991), and the concat stitch fails with an opaque moov-atom error — permanently, until the user manually deletes the hash-named file or runs --force (full re-render of ALL scenes). In the SIGTERM-graceful variant (ffmpeg writes the trailer before the 3s SIGKILL), the truncated-but-playable scene can ship silently into final.mp4 if the duration delta stays under review.ts's 5% warn threshold. GC (pruneStaleSceneArtifacts) never reclaims the poisoned file because its hash is still live. Fix is the temp-name + renameSync atomic-publish pattern already documented in fsutil.ts.
- 修法:Render/encode into a temp name inside the scene working dir (e.g. sceneDir/out.part.mp4) and fs.renameSync onto mp4Path only after the subprocess resolves — the same atomic-publish pattern as writeFileAtomic. Two-line change per engine path.
